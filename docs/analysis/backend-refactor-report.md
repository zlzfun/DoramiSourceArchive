> 面向读者：想快速了解「这次后端重构改了什么、为什么、效果如何」的人。
> 配套：诊断见 [`backend-architecture-review.md`](backend-architecture-review.md)、路线见
> [`backend-refactor-plan.md`](backend-refactor-plan.md)、逐项进度见
> [`backend-refactor-progress.md`](backend-refactor-progress.md)。
> 日期：2026-07-03 · 分支 `refactor/phase0-concurrency` · 测试 249 → **304**（全程全绿）。

---

## 一、一句话概括

把一个「**5354 行的 API 巨石 + 全局内存状态 + 长任务卡在请求里 + 手写无版本迁移**」的后端，
重构为「**按域拆分的 15 个 Router + 依赖注入 + 持久化任务状态机 + 版本化数据库迁移 + 跨存储对账 +
启动安全校验**」的结构，**全程不改变任何对外接口路径、不破坏任何测试**。

---

## 二、重构前的四类结构性债务

| 债务 | 具体表现 | 影响 |
|---|---|---|
| **API 巨石** | `app.py` 5354 行、109 个路由全部内联；全局单例 `db_sink/vector_sink/...`；手开 `Session` 109 次 | 难维护、难测试、改一处牵全身 |
| **状态在内存** | 后台任务、抓取进度、日报进度都存在进程内 dict | 重启即丢、无法多实例、无法断点续查 |
| **长任务即请求** | 全量向量化、重索引、日报、批量抓取都在 HTTP 请求里 `await` 数分钟 | 占满连接、易超时、体验差 |
| **数据层脆弱** | SQLite 靠 `create_all` + 手写 `ALTER TABLE` 迁移（无版本、无回滚、漏建索引）；SQLite 与 ChromaDB 靠顺序调用维持一致、无对账 | schema 演进无保障、两套存储会悄悄漂移 |

## 三、架构：重构前 → 重构后

### 重构前

```
                     ┌──────────────────────────────────────────────┐
                     │           src/api/app.py  (5354 行)           │
   HTTP 请求 ───────▶│                                              │
                     │  · 109 个路由全部内联在一个文件                 │
                     │  · 全局单例 db_sink / vector_sink / scheduler  │
                     │  · 权限：中间件 + 前缀表（新增端点易漏审）       │
                     │  · 长任务在请求内 await（分钟级阻塞连接）        │
                     │  · 任务/进度状态：内存 dict（重启即丢、单进程）   │
                     └───────────────────┬──────────────────────────┘
                                         │  手开 Session(db_sink.engine) × 109
                       ┌─────────────────▼──────────────────┐
                       │  SQLite  ── create_all + 手写 ALTER  │  ← 无版本/无回滚/漏建索引
                       │  ChromaDB ── 与 SQLite 顺序调用维持一致 │  ← is_vectorized 布尔、会漂移、无对账
                       └────────────────────────────────────┘
```

### 重构后

```
                     ┌──────────────────────────────────────────────┐
                     │       src/api/app.py  (1677 行 · 组装层)       │
                     │  · app.include_router × 15                     │
                     │  · 中间件鉴权 + 「鉴权覆盖审计」测试护栏         │
                     │  · 采集引擎 / APScheduler / lifespan           │
                     │  · 启动安全校验（生产姿态拒绝弱 secret / 危险 CORS）│
   HTTP 请求 ─┬─────▶└───────────────────┬──────────────────────────┘
              │        15 个域 Router（articles / vector / reader /
              │        collection / daily_brief / source_configs / ...）
              │        ── 统一经 deps.py 依赖注入拿 session / sink
              │
              │   长任务 ──▶ jobs.launch() 立即返回 job_id
              │              └─ services/jobs.py（JobRecord 状态机）
              │                 提交→运行→终态全部落库、可查、重启不丢
              │              前端轮询 GET /api/jobs/{id}
              ▼
     ┌───────────────────────┐   对账   ┌──────────────────────────────┐
     │ SQLite（Alembic 迁移）  │◀──────▶│ ChromaDB                       │
     │ · 版本化 / 可回放       │  巡检   │ vector_reconcile：            │
     │ · index_status 状态枚举 │        │  按需端点 + 每日 04:00 定时巡检 │
     └───────────────────────┘        └──────────────────────────────┘
```

**关键结构变化一目了然**：单文件 → 组装层 + 15 Router；内存状态 → 落库状态机；请求内长任务 →
提交即返回 + 轮询；`create_all`+手写 ALTER → Alembic 版本化迁移；布尔 + 无对账 → 状态枚举 + 主动对账。

## 四、改动点详解（按阶段）

### 阶段 0 · 并发止血（前序会话完成）
- 向量 `save/search/rerank`、重索引等**同步重计算卸载到线程池**，不再冻结事件循环。
- 生产默认关闭 `reload`；SQLite 开启 **WAL + busy_timeout**（读写并发不再互锁）。

### 阶段 1 · 结构拆分（前序会话完成）
- **把 `app.py` 从 5354 行拆到 1677 行**：按业务域抽出 15 个 `APIRouter`（文章、向量、阅读器、
  采集、日报、数据源、抓取器、归档同步、订阅……），**所有端点路径保持不变**。
- 抽出 `api/deps.py` 依赖注入（`get_session` / `get_db_sink` / `get_vector_sink`），取代到处手开
  `Session`。
- 新增**「鉴权覆盖审计」测试**：遍历所有 `/api/*` 路由，强制每条都落入明确的权限分类——
  这样**任何新增端点若漏配权限闸，测试会立刻失败**，把"漏审"从隐患变成红灯。

### 阶段 2 · 数据层固化（本轮）
- **引入 Alembic 版本化迁移**：以当前线上 schema 生成基线；`ensure_migrated()` 能把「有表但从未
  纳入 Alembic」的老库先打基线戳、再升级（避免重跑建表）；`deploy.sh` 在启动后端前自动迁移。
  一条**漂移守卫测试**锁死「`create_all`（模型定义）永远等于迁移链」，杜绝改了模型漏写迁移。
- **修复旧库缺失的索引**：老的手写 `ALTER TABLE ADD COLUMN` 只加列不建索引，导致模型上声明了
  `index=True` 的列在旧库其实没有索引——一条幂等对账迁移把它们补齐。
- **SQLite ↔ ChromaDB 对账**（`vector_reconcile`）：比对「数据库以为已向量化的」与「向量库里实际存在
  的」，分出三类漂移——**丢索引 / 未标记 / 孤儿 chunk**，可只报告也可一键修复。提供管理员端点
  `GET/POST /api/vector/reconcile`，并注册**每日 04:00 定时巡检**（只报告、发现漂移告警）。
- **`is_vectorized` 布尔升级为 `index_status` 状态枚举**（`pending/indexing/indexed/failed/stale`）：
  能区分「从未索引 / 索引中 / 已索引 / 失败 / 内容改动待重建」。采用**附加式**——保留 `is_vectorized`
  作为向后兼容派生位，旧的过滤参数和前端零改动；知识台账新增状态徽标（失败/陈旧带语义色）与按状态筛选。

### 阶段 3 · 任务化（本轮）
- **持久化任务状态机 `services/jobs.py`（`JobRecord` 表）取代进程内内存版**：状态
  `queued/running/succeeded/failed/cancelled` 全部落库，进度节流写入，**进程重启不丢、可跨进程查询**。
- **五类长任务全部改为「提交即返回 job_id + 前端轮询」**：全量向量化、全量重索引、每日日报（手动生成）、
  批量抓取活跃数据源、采集任务运行。请求不再一卡几分钟；前端统一用 `pollJob` 拿最终结果，细粒度进度
  仍走各自的进度端点。校验（如任务不存在）仍在提交前同步返回，行为不变。

### 阶段 4 · 编排下沉 + 安全硬化（本轮）
- **启动安全校验 `api/security_checks.py`**：按部署姿态分级——**生产姿态**（`cookie_secure=true`，
  即走 HTTPS）下，若 `secret` 未设/仍是占位符、或 CORS `allow_origins=* 且 allow_credentials=true`，
  **直接拒绝启动**；开发姿态仅告警不阻断。
- **AUTH_SECRET 不再从口令派生**：缺省回退去掉了明文种子口令，仅用 `database_url` + 固定盐派生。
- **阅读器三档问答上下文下沉**：把「当前文章 / 订阅 RAG 召回 / 最近订阅文章兜底」的组装逻辑从端点
  内联抽成可复用、可独立单测的 `reader_ai.assemble_reader_context`（用注入的取数闭包解耦 HTTP 请求）。

## 五、量化效果

| 指标 | 前 | 后 |
|---|---|---|
| `app.py` 行数 | 5354 | **1677**（−69%） |
| 域 Router 数 | 0（全内联） | **15** |
| 数据库迁移 | 手写 ALTER、无版本 | **Alembic 版本化**（基线 + 3 迁移） |
| 长任务返回方式 | 请求内 await 数分钟 | **提交即返回 job_id**，状态落库可查 |
| 向量状态 | 布尔、无对账 | **5 态枚举** + 按需/定时对账 |
| 生产弱配置 | 静默启动 | **启动即拒绝**（secret/CORS） |
| 自动化测试 | 249 | **304**（+55，全绿） |

## 六、有意「暂缓/舍弃」的项（附理由）

重构遵循「只做有真实价值的改动」，以下项**经评估后明确不做**（详见进度文档各表备注）：

- **应用工厂化 / 权限迁到 route dependency**：安全实质（漏审防护）已由审计测试护栏收口；全量迁移是
  风格性改动、有回归风险。
- **`read_auth_token` 加缓存**：每请求校验数据库**正是**「禁用/删除账户即时吊销会话」的安全特性，
  缓存反而会削弱它——当前即最优。
- **外键 / JSON 提列 / 唯一约束**：SQLite 下多为装饰性或需重建表，收益低。
- **进度落库 + scheduler/worker 拆分**：其价值与「多实例部署」强绑，当前单节点无需求；`jobs` 表已为
  将来拆分预留持久化地基。
- **全量 metrics/trace、密钥迁 secret manager、fetcher manifest 化**：低优先运维项。

这些均**非阻塞**，将来出现多实例部署或具体需求时，可照进度文档记录按需重启，地基已铺好。
