设计推导:调度台(dispatch desk)

运行页只需回答三个问题:什么在跑(总账条「运行中」灯 + 流水首行进度线)、什么时候跑(左栏时刻表:按下次触发排序 + 倒计时走字)、跑得怎么样(按日分组的运行流水)。仪器感来自持续走字的台面时钟与倒计时——功能性的运动,不是装饰动效。

复用的既有范式(与台账/节点同基因)

  • 页头 .page-head(24/800 + 右侧动作区,CTA 唯一实底 accent);自动刷新 = 节点页 signal-autorefresh 同刻度。
  • 总账条 = 台账 ledger-strip 直移:28/700 tabular 读数,前五格可点 = 状态筛选(选中 = 底部 accent 条),「新增入库」为被动读数;右端工具区 soft 底 + .mini-seg(触发/对象二择)。
  • 选中语法 R1(时刻表任务行 = accent 竖条 + soft 底)、状态章 .stamp、页码 .pager、失败就地展示(节点页范式)、行内进度线(活动态 = accent 家族,产品语义)。
  • 任务上下文条 = 台账批量条语法(wash = 状态标记底):选中任务即过滤流水,任务级动作(立即运行/测试/编辑/停用)就地给出,不再需要「采集任务/运行历史」两视图切换。

信息设计决策

  • 抓取/新增/跳过拆成三个右对齐 mono 列(账本对齐),「新增」是主读数(ink/700),零值退为「–」;原「抓取 24 / 新增 18 / 跳过 6」句式列拆掉。
  • 日分组行 = 日期导轨 + 当日摘要(N 次 · 新增 · 失败),soft 底压一档;聚合运行行内展开子节点(well 底细行),失败子节点错误贴行展示。
  • 时刻表行三层:名称 + 上次结果形状点 / 节点数 + mono cron / 7 日点阵 + 下次倒计时。停用任务压 opacity 收进「已停用」组。

临时运行(未存任务)的处理

  • 原则:时刻表 = 承诺清单,只陈列有「下次」的对象;临时运行(节点页直接执行/批量运行,run_scope=ad_hoc)没有未来,主体归流水,不与任务混排。
  • 固定汇总位:时刻表底部「未存为任务」组一枚「临时抓取」汇总行(今日次数 + 7 日点阵,无倒计时),选中即过滤右侧流水——与工具区「对象:临时」mini-seg 是同一筛选的双入口。
  • 升格回路:临时运行行尾 hover 浮现「存为采集任务」(台账 rowacts 范式,预填该次运行的节点 + 参数进任务编辑器)——与节点页批量选择的「存为采集任务」同一动作、两个入口;反复手动跑的组合据此升格为时刻表居民。失败行同位给「重跑」。
  • run_scope 映射:saved_job = 任务 · ad_hoc = 临时(单节点手动与批量临时都归此)· legacy_task = 旧版,展示上并入临时组。

实装依赖与取舍(诚实标注)

  • 「下次触发倒计时」需后端暴露 next_fire_time(APScheduler 有此数据,现 API 未透出);7 日点阵与当日摘要需 per-day 聚合端点——与台账 sparkline 同一遗留立项。
  • 聚合运行的子节点展开:数据已有(FetchRunRecord.job_run_id 关联),仅需列表接口支持按 job_run_id 取子运行。
  • 流水行「存为采集任务」需从运行记录回读节点集 + 生效参数预填草稿(子 FetchRunRecord 有 fetcher_id/params,可拼装;与节点页 pendingJobDraft 复用同一编辑器入口)。
  • 任务编辑仍走既有模态(节点选择/参数覆盖/单节点 cron),样页未含;「两视图 segmented」被「时刻表 + 上下文条」替代,是本设计最大的结构变更,需产品确认。
